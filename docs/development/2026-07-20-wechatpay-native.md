# 微信支付 Native 接入与运维

## 功能范围

本实现用于钱包充值，采用微信支付 API v3 的 Native 支付模式：服务端创建预支付单，网页展示微信 `code_url` 二维码，用户扫码支付，服务端通过加密通知与主动查单确认结果后入账。

当前不包含 JSAPI、小程序、App、H5 支付、退款和订阅购买。

为降低 fork 合并上游时的冲突，微信支付保持为独立 provider：新增独立的 setting、service、controller、router、pending task 与两套前端组件；对既有公共文件只增加路由挂载、支付方式枚举、迁移和页面入口等少量接缝。不改动已有支付宝协议和订单逻辑。

## 处理流程

1. 已登录用户调用 `POST /api/user/wechatpay/native` 创建订单。
2. 服务端校验后台配置的单笔上下限（最高不超过 4000 个计费单位），以有界十进制运算计算人民币实付金额和整数“分”，并在调用微信前检查用户额度是否仍有可入账空间。
3. 服务端在一个数据库事务中保存 `top_ups` 与 `wechat_pay_pending_tasks`，前端展示 `code_url` 二维码。
4. 微信向 `POST /api/wechatpay/notify` 推送加密通知；服务端先验签、再解密，然后校验 `appid`、`mchid`、本地单号、交易类型、金额和币种。
5. 入账事务只允许 `wechatpay` / `wechatpay_native` 的 pending 订单变为 success，并在同一事务中以条件更新增加用户额度。重复通知、查单并发或额度同时变化时不会重复入账或产生整数溢出。
6. 数据库提交后失效用户额度缓存，下次读取以数据库结果回填，避免“缓存增量更新”与“绝对值回填”竞态造成重复显示。
7. 主节点后台任务通过数据库租约认领未完成订单。通知丢失时可由查单补偿；到期后先向微信关单，再更新本地状态。
8. 前端只轮询本地 `GET /api/user/order/status?type=topup&trade_no=...`，不让浏览器直接频繁调用微信查单接口。

## 上线前准备

### 1. 微信商户平台

准备以下信息：

- 与商户号绑定的应用 `AppID`
- 微信支付商户号 `mchid`
- 商户 API 证书序列号
- 对应证书的 PKCS#8 RSA 商户私钥 PEM
- 32 字节 APIv3 密钥
- 微信支付公钥 ID（通常以 `PUB_KEY_ID_` 开头）
- 微信支付 RSA 公钥 PEM

本实现使用微信支付公钥模式，不使用平台证书自动下载模式。公钥 ID 和公钥必须属于同一份当前有效的微信支付公钥。微信支付公钥不是商户 API 证书中的公钥，系统会拒绝把商户私钥对应的公钥误填为微信支付公钥。

参考官方文档：

- [Native 下单 API](https://pay.wechatpay.cn/doc/v3/merchant/4012791877)
- [Native 支付开发指引](https://pay.wechatpay.cn/doc/v3/merchant/4012791891)
- [支付通知与查单](https://pay.wechatpay.cn/doc/v3/merchant/4012075249)
- [支付通知处理规则](https://pay.wechatpay.cn/doc/v3/merchant/4012075420)
- [官方 Go SDK](https://github.com/wechatpay-apiv3/wechatpay-go)

### 微信支付没有独立沙盒

微信支付 API v3 没有类似支付宝沙盒的独立测试环境。联调时应使用受限测试账号、专用测试商品或业务分组以及小额真实支付，并完成支付、通知、查单、关单和对账验证。不要对生产支付接口做并发压测；并发与幂等测试应在本地使用模拟通知和数据库测试完成。退款不在当前实现范围内，真实联调款项需要按商户自己的流程处理。

### 2. 服务端环境变量

在所有应用节点配置相同的 `OPTION_CRYPT_KEY`：

```dotenv
OPTION_CRYPT_KEY=使用密钥管理系统生成并注入的高熵随机值
```

该值用于加密数据库中的商户私钥、APIv3 密钥和微信支付公钥。它不是 APIv3 密钥，二者不可复用。不要把真实值提交到 Git、镜像、日志或工单。

如果多节点的 `OPTION_CRYPT_KEY` 不一致，部分节点将无法解密支付配置；如果丢失该值，已加密配置无法恢复，只能重新录入/轮换支付密钥。

### 3. 外网回调

回调地址必须满足：

- 公网可访问的 HTTPS 地址；
- 不带用户名、密码、query 或 fragment；
- 反向代理不得改写微信验签所需的请求头或原始请求体；
- 路径默认是 `https://你的站点/api/wechatpay/notify`。

若部署环境自动推导的站点地址不正确，可在管理后台显式填写通知地址。该路径不需要用户登录，但每个请求都必须通过微信支付 RSA 签名验证和 APIv3 解密。

## 管理后台配置

使用 root 管理员进入「设置 → 支付设置 → 微信支付」，填写：

| 字段 | 说明 |
|---|---|
| 启用微信支付 | 只控制新订单入口；关闭后已有 pending 订单仍会接收通知和补偿查单 |
| AppID | 与商户号绑定的应用 ID |
| 商户号 | 微信支付 `mchid` |
| 商户证书序列号 | 商户 API 证书序列号，不是微信支付公钥 ID |
| 商户私钥 | PKCS#8 RSA PEM；留空保存表示保留现值 |
| APIv3 密钥 | 必须正好 32 字节；留空保存表示保留现值 |
| 微信支付公钥 ID | 当前公钥对应的 `PUB_KEY_ID_*` |
| 微信支付公钥 | RSA PUBLIC KEY PEM；留空保存表示保留现值 |
| 通知地址 | 可留空使用系统默认地址 |
| 最低充值数量 | 必须大于 0 |
| 最高充值数量 | 不小于最低值，且不得超过服务端硬上限 4000 个计费单位 |
| 订单有效分钟数 | 1–120 分钟 |

保存成功后，管理接口只返回密钥是否已配置，不会回显密钥原文；前端也会立即清空刚提交的密钥文本，即使后续配置回读失败也不会继续显示。配置首次加载失败时保存按钮保持禁用，避免用空表单覆盖有效配置。

清除密钥必须使用界面的显式清除操作；为避免误删，提交空输入不会覆盖现有密钥。清除任意必要密钥时系统会同时关闭微信支付新订单入口，并使依赖旧密钥的通知验签和后台查单不可用。存在 pending 订单时服务端默认拒绝清除；`force_clear_secrets=true` 仅用于密钥已经泄露、必须紧急吊销的场景，操作前应先完成对账并保留审计记录。

## 安全设计

- 使用微信支付官方 Go SDK 构造请求、验签并解密通知，不手写签名协议。
- 私钥、APIv3 密钥和微信支付公钥以 AES-GCM 加密后存入既有 `options` 表，明文只保留在进程内存。
- 支付配置管理接口只允许 root 管理员访问，并受关键操作限流保护。
- 通知请求体最大 64 KiB，配置请求体最大 256 KiB，防止无界读取。
- 不记录通知原文、签名头、私钥、APIv3 密钥或解密后的完整交易数据。
- 商户订单号使用密码学随机数，长度不超过微信限制。
- 金额统一在服务端计算并转为整数分；通知金额必须与本地订单完全一致，币种必须为 `CNY`。
- 金额、充值数量和额度转换拒绝 NaN、无穷值、非正数及超出 `int64` / 跨数据库 `INT` 范围的结果；最高充值配置还有 4000 的独立硬上限。
- 下单前执行额度容量预检；入账事务再次使用 `quota <= MaxInt32 - 增量` 条件更新。余额更新和订单成功状态处于同一事务，支持 SQLite、MySQL 和 PostgreSQL。
- 支付配置写入先锁定统一的数据库配置行，以最新完整配置在同一事务内校验和提交；单个进程再以一次加锁批量发布，读取方只能看到完整旧快照或完整新快照，不会看到一半新一半旧的密钥组合。
- 待处理任务使用数据库租约和租约所有者条件更新；进程崩溃后租约会自动到期，旧所有者不能删除或延期已被其他节点重新认领的任务。
- 前端二维码来自后端已验证的微信下单响应；订单状态查询仅返回当前登录用户自己的订单。

## 停用和密钥轮换

普通停用只需关闭「启用微信支付」。不要立即清除旧密钥：已有 pending 订单仍需要旧配置完成通知验签和查单。

推荐轮换步骤：

1. 关闭新订单入口。
2. 在后台确认 pending 订单数量为 0；必要时等待最长订单有效期结束。
3. 在微信商户平台生成/启用新密钥或公钥。
4. 在后台一次性更新相互匹配的序列号、私钥、公钥 ID、公钥和 APIv3 密钥。
5. 保存并重新启用，先用小额真实订单完成端到端验证。
6. 确认新订单通知和主动查单正常后，再在微信平台撤销旧材料。

修改 `OPTION_CRYPT_KEY` 时不能直接覆盖环境变量。应先在旧密钥可用时重新加密所有敏感 Option；当前版本未提供自动主密钥轮换命令，因此若确需轮换，应在维护窗口备份数据库、关闭支付入口、清除并重新录入支付密钥，然后逐节点更新并验证。

## 多节点部署

- 所有节点必须共享数据库，并使用相同的 `OPTION_CRYPT_KEY` 和微信支付配置。
- 所有节点均可接收通知；幂等入账保证重复投递安全。
- 仅 `NODE_TYPE=master` 的节点运行主动查单任务；数据库租约为误配置多个 master 的情况提供额外互斥保护。租约当前为 2 分钟，任务每 30 秒扫描一次，每次只认领即将处理的一条（每轮最多 100 条），避免串行队列中的租约提前过期。单次微信请求超时为 15 秒；正常 pending 订单 30 秒后复查，网络、结算或关单错误采用 30 秒到 30 分钟的指数退避。
- 每个节点的运行时配置按完整快照发布，但节点之间仍依赖现有 Option 同步周期，不能把多节点配置更新视为瞬时一致。密钥轮换应关闭新订单、清空 pending、一次保存完整配置，并等待或重启各节点完成同步后再恢复入口。
- 回调负载均衡不能对请求体做 JSON 重排、解压后重编码或请求头过滤。

## 验收清单

1. 管理接口与普通 Option 列表均不会返回密钥原文。
2. 数据库对应敏感 Option 值以 `enc:v1:` 开头，而不是 PEM/APIv3 明文。
3. 用户充值页可展示二维码，二维码过期后不能继续支付。
4. 支付成功后余额只增加一次，充值记录状态为 success，日志包含微信交易号但不包含密钥。
5. 暂时阻断回调后，主动查单能够补偿入账。
6. 伪造签名、错误商户号、错误金额、错误币种和其他支付渠道订单均不能入账。
7. 关闭新订单入口后不能创建新单，但已付款 pending 单仍能完成入账。
8. 在 SQLite、MySQL 和 PostgreSQL 环境至少分别验证迁移；生产数据库升级前先备份。
9. 使用小额真实支付完成上线前端到端验收；不要寻找不存在的独立沙盒，也不要对生产微信支付 API 做压力测试。

## 常见问题

### 创建订单失败

检查 AppID 与商户号绑定关系、商户证书序列号是否对应当前私钥、服务端时间是否同步、出口网络是否可访问微信 API，以及 APIv3 密钥长度是否正好为 32 字节。应用日志只记录本地单号和概括错误，不会打印密钥。

### 扫码成功但未入账

先在管理后台查看 pending 数量，再检查回调地址的公网 TLS、反向代理请求头透传、主节点任务是否运行，以及所有节点的 `OPTION_CRYPT_KEY` 是否一致。不要手工修改用户余额；修复配置后让通知重试或主动查单补偿。

### 回调持续返回 FAIL

常见原因是微信支付公钥 ID 与公钥不匹配、APIv3 密钥错误、原始请求体被代理改写、订单金额或商户信息不一致。服务端故意对校验失败返回非 2xx，让微信按官方规则重试。

### 后台显示密钥已配置但支付不可用

可能是节点启动时无法用 `OPTION_CRYPT_KEY` 解密数据库中的值。恢复原主密钥并重启节点，或在受控维护窗口清除并重新录入密钥。
