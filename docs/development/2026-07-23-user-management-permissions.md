# 用户管理权限

系统保留原有四种角色：游客（`0`）、普通用户（`1`）、管理员（`10`）和超级管理员（`100`）。在角色之外，超级管理员可以为指定的普通用户授予细粒度管理权限，无需把用户提升为管理员。

## 权限清单

| 权限标识 | 能力范围 |
| --- | --- |
| `skill_hub.content.manage` | 技能、标签、资源上传及技能发布/下架 |
| `skill_hub.reports.manage` | 技能广场举报查看与处理 |
| `chat_models.manage` | 对话模型配置 |
| `client_releases.manage` | 客户端版本创建、编辑、上传与删除 |
| `client_releases.publish` | 客户端版本发布与取消发布 |

管理员和超级管理员隐式拥有全部管理权限。显式权限只允许授予普通用户；游客、管理员和超级管理员不会保存显式权限。用户被提升或降级时，历史显式权限会在同一事务中清理，防止后续角色变化意外恢复旧授权。

## 管理接口

```text
GET /api/user/{id}/management-permissions
PUT /api/user/{id}/management-permissions
```

两个接口都仅允许超级管理员访问。`PUT` 与现有用户管理接口保持一致，不强制要求两步验证或 Passkey；写操作仍受关键接口频率限制。请求体必须显式提供完整权限集合：

```json
{
  "permissions": [
    "chat_models.manage",
    "client_releases.publish"
  ]
}
```

空数组表示撤销全部显式权限。未知权限、非普通用户目标、非法用户 ID 和超限请求都会被拒绝。授权成功后会写入管理日志，包含操作者、变更前权限和变更后权限。

## 鉴权与即时撤销

- 受保护接口先执行登录鉴权，再从数据库读取用户当前状态、角色和显式权限。
- 权限判断不依赖浏览器菜单、Session 中的旧角色或长时间缓存；禁用用户和权限撤销会在后续请求中立即生效。
- 前端路由和菜单只负责体验优化，后端中间件始终是最终授权边界。
- `/api/user/self` 和登录响应返回 `management_permissions`，供 default/classic 两套前端过滤菜单和路由。

## 并发与数据一致性

- 权限集合采用“全量替换”语义，并在数据库事务中执行。
- MySQL 和 PostgreSQL 会锁定目标用户行；SQLite 依赖写事务串行化。角色变更、权限替换和用户删除共享同一用户行同步点。授权事务还会校验并锁定操作者用户行，只有数据库中仍处于启用状态的超级管理员可以提交授权。
- 权限替换不会产生两个请求权限集合的并集；最终结果是某一次完整请求。
- 用户软删除或硬删除时会在同一事务中清理显式权限。
- 客户端版本元数据修改与发布状态使用独立权限。仅版本管理权限可以维护草稿；修改或删除已发布记录还需要发布权限。事务会先锁定操作者用户行并重新读取有效权限，再锁定版本行；因此撤权、角色变更与版本写入有确定的串行顺序。元数据更新在行锁内保留数据库中的真实发布状态；发布/下架也在行锁事务中更新，避免并发请求用旧对象覆盖状态或其他字段。

## 双前端行为

- default 和 classic 的用户编辑界面仅对超级管理员显示权限配置。
- 普通用户只会看到自己获准的管理入口；直接访问未授权路由会进入无权限页面。
- 对话模型委派用户只能看到“对话模型”页签，不会触发模型广场或部署管理接口。
- 客户端版本编辑和发布按钮按两个独立权限显示；仅有发布权限的用户只能查看记录并发布或下架。
