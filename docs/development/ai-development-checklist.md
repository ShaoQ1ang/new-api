# AI 开发前必读与收尾清单

本文档是 AI 参与 `z-up-new-api` 开发前必须遵守的工作规则。每次写代码、改代码、排查问题或处理评审意见前，都要先按这里执行。

## 1. Web 双前端同步

本仓库 Web 前端分为两套：

- `web/classic`：classic 前端。
- `web/default`：default 前端。

凡是涉及 Web 功能、页面、组件、样式、交互、接口适配、状态管理、国际化、构建配置或运行时行为的变更，都必须同时检查两套前端：

- 改 `web/classic` 时，要检查 `web/default` 是否需要同步更新。
- 改 `web/default` 时，要检查 `web/classic` 是否需要同步更新。
- 用户可见行为、接口契约、权限判断、错误处理、空状态、加载状态和文档说明应尽量保持一致。
- 如果某次改动只适用于其中一套，必须在相关开发文档或交付说明中写明原因。

## 2. 开发后整理文档

代码开发完成后，必须同步整理相关开发文档。

常见需要更新的内容：

- API、OpenAPI、请求/响应 DTO、错误码或鉴权逻辑变化。
- 前端页面、配置项、国际化、操作流程或可见行为变化。
- 部署、环境变量、数据库迁移、缓存、队列、定时任务或自动更新流程变化。
- 计费、渠道、支付、客户端发布、技能广场等业务模块变化。
- 安全边界、权限规则、限制条件、排障步骤或已知风险变化。

文档位置按影响范围选择：

- `README*`：项目入口、安装和常用运行方式。
- `docs/`：专题设计、开发记录、接口说明和排障说明。
- `AGENTS.md` / `CLAUDE.md`：AI 必须长期遵守的开发约定。
- 模块附近文档：只影响单一模块的实现细节。

## 3. 开发后风险复查

开发完成后，必须主动复查安全问题、并发问题和其他明显缺陷。

安全复查至少覆盖：

- 是否泄露 token、api key、cookie、密钥、本地路径或用户敏感信息。
- 鉴权、权限、用户隔离、管理员能力和二次验证是否完整。
- 输入参数、文件路径、外部 URL、回调地址、webhook、支付参数是否校验。
- 是否引入 SQL 注入、命令注入、XSS、SSRF、路径穿越、任意文件读写或越权访问风险。
- 前端是否把敏感信息持久化到不合适的位置。

并发复查至少覆盖：

- 是否存在重复请求、重复事件监听、重复定时器、重复 goroutine 或重复后台任务。
- 异步结果是否可能乱序覆盖新状态。
- 是否正确处理取消、超时、重试、幂等、锁、事务和资源释放。
- 支付、计费、额度、用户状态、客户端更新等流程是否能抵抗重复提交和竞态。
- SQLite、MySQL、PostgreSQL 三种数据库下行为是否一致。

其他质量复查至少覆盖：

- 错误路径、空状态、边界输入、分页、排序、筛选和权限不足状态是否可用。
- default/classic 前端行为是否一致，或已说明不一致原因。
- 类型、schema、默认值、迁移、文档和 OpenAPI 是否一致。
- 是否需要运行 Go 测试、前端 typecheck/build、OpenAPI 生成或更具体的验证命令。

## 4. 交付前简短确认

交付时应确认：

- 已检查 `web/classic` 和 `web/default` 的影响范围。
- 已更新相关开发文档。
- 已复查安全、并发和明显缺陷。
- 已运行与改动风险匹配的检查，或说明未运行的原因。
